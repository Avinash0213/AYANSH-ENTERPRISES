Issues & Proposed Improvements

Current structural problems and recommended changes for your upcoming refactor.

🔧
Business Logic In Controllers
HIGH
CustomersController directly queries the DB via AppDbContext instead of going through a service layer. The inline LINQ for filtering (search, type, dateFilter) and the manual CustomerResponse mapping in every action creates a lot of duplication and breaks the single-responsibility principle.
💡 Fix
Move all DB queries into CustomerService. Create a ICustomerRepository or extend CustomerService with GetAllAsync(filter), GetByIdAsync(id). Controllers should only call services and map results.
🗺️
No DTO Mapping Layer
HIGH
Every controller action manually builds CustomerResponse objects field-by-field. This means if you add a field to the entity, you must update 4 different places (GetAll, GetById, Create, Update). Very fragile and repetitive.
💡 Fix
Introduce AutoMapper or a simple static CustomerMapper.ToResponse(Customer c) extension method to centralize mapping. One change propagates everywhere.
💾
Mixed Repository Access Patterns
MEDIUM
Some controllers inject AppDbContext directly while others use services. There is a GenericRepository that isn't being used anywhere in the controllers. This inconsistency makes auditing and testing hard.
💡 Fix
Enforce a clear layering rule: Controllers → Services → Repository → DbContext. Remove direct DbContext injection from controllers. Either use the generic repo or per-entity repos.
📅
UTC vs Local Time Inconsistency
MEDIUM
The dateFilter logic in CustomersController compares DateTime.UtcNow with stored CreatedDate values, but it doesn't verify that CreatedDate is stored as UTC consistently across all creation paths.
💡 Fix
Always use DateTime.UtcNow for storage and enforce it in a base entity class or EF Core interceptor. Add [Column(TypeName = "timestamp with time zone")] in PostgreSQL mappings.
🔒
Missing Permission on Customer Endpoints
MEDIUM
All Customer endpoints only require [Authorize] (JWT) but have no [RequirePermission] attribute. Any logged-in user (even with no permissions) can create, update, or delete customers.
💡 Fix
Add permission keys like CUSTOMER_VIEW, CUSTOMER_CREATE, CUSTOMER_UPDATE, CUSTOMER_DELETE and apply [RequirePermission] to each action accordingly.
🧪
No Soft Delete on Payments or Users Queried
LOW
The User entity has IsDeleted, but it's not clear if EF Core global query filters are applied to automatically exclude soft-deleted records. The Payment entity has no soft delete at all.
💡 Fix
Consider adding a HasQueryFilter(u => !u.IsDeleted) in DbContext for User. For Payments, decide if soft-delete is needed for audit trails. Consider adding a BaseEntity with IsDeleted and a query filter applied via reflection.
⚡
Profit Computed on Every Save
LOW
Payment's Profit is stored as a computed column in the database entity. If the formula changes, historical records become incorrect. Currently it's computed in PaymentService before saving.
💡 Fix
Either mark Profit as a calculated property (not stored) and compute it at query time via projection, or add a clear data contract that Profit = ReceivedAmount − GovernmentCharges − EmployeeCommission and document it.
🏗️
Single Renewal Endpoint
LOW
RenewalsController has only one GET endpoint. There's no way to record a renewal, extend a contract, or track renewal history. Renewals are read-only and derived from Customer EndDate.
💡 Fix
Add a Renewal entity to track renewal events. Add POST /api/renewals to record when a renewal occurred and GET /api/renewals/history/{customerId} for audit history. Update Customer.EndDate upon renewal.

COST-SAVING TIPS
🌙 Keep Railway Costs Low
Railway bills per second of compute. Since your API auto-starts on request, make sure it's not running unnecessary background jobs. The $5 free credit covers ~500 idle hours — more than enough for a small BMS with occasional use.
💤 Neon Auto-Suspend
Neon's free tier auto-suspends the DB after 5 minutes of inactivity. This is fine — it will wake up in ~1–2 seconds on the next request. Your app's first request after idle may be slightly slower — this is expected.
📦 Keep DB Lean
Neon free gives 512 MB. For a property management BMS, that's enough for thousands of customers and payments. Monitor size via Neon dashboard. Export and archive old records locally before hitting limits.
🔒 Secrets Management
Never commit appsettings.json with real DB passwords to GitHub. Use Railway ENV vars for all secrets. Add appsettings.Development.json to .gitignore if it contains real credentials.
🎯
Bottom line: For 2 years, you can realistically run this BMS for under $25 total (just the domain) if DB stays under 512 MB and API usage remains moderate. When you're ready to move up, Railway → Render/Azure App Service and Neon → Supabase Pro or Azure Database for PostgreSQL are smooth migration paths with minimal code changes.

Configure NGINX as Reverse Proxy NGINX
NGINX will handle SSL termination and forward requests to your API (port 5000) and serve the React build (static files).
NGINX Config (/etc/nginx/sites-available/bms)
server {
  listen 80;
  server_name yourdomain.com www.yourdomain.com;

  # Serve React frontend (static build)
  root /var/www/bms-web;
  index index.html;

  location / {
    try_files $uri $uri/ /index.html;
  }

  # Proxy API requests to .NET Core
  location /api/ {
    proxy_pass http://localhost:5000;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
  }
}

# Enable config & test
